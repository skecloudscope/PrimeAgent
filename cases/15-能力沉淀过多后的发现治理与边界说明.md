# 15 能力沉淀过多后的发现治理与边界说明

## 文档目的

本文件回答一个长期会非常现实的问题：

如果跨境电商平台里沉淀了大量 Skill、Tool、MCP / Connector、Agent、Team、Workflow Skeleton、Workflow，系统会不会找不到？AI 能不能自己找到过去已经沉淀的好能力？每个 Skill / MCP / Tool 是否必须有说明，讲清楚能力边界？

核心结论：

```text
必须建立 Capability Registry。

所有 Skill / Tool / MCP / Agent / Team / Workflow Skeleton / Workflow 都必须注册成 Capability。
每个 Capability 必须有清晰说明、适用场景、输入输出、边界、风险等级、权限要求、质量指标和示例。
Orchestrator 不应该凭 prompt 记忆找能力，而应该通过 Capability Retrieval 找能力。
```

## 一句话结论

能力沉淀越多，越不能只靠名字和人工记忆。

平台必须支持：

- 能力卡片。
- 能力标签。
- 能力边界说明。
- 输入输出 schema。
- 适用/不适用场景。
- 风险等级。
- 权限和数据源要求。
- 版本和状态。
- 质量评分。
- 使用次数和成功率。
- 语义检索。
- 规则过滤。
- LLM rerank。
- 用户确认。
- 重复能力合并。
- 过期能力下线。

## 为什么会找不到能力

跨境电商场景会快速产生大量能力：

- 竞品链接分析 Skill。
- Amazon 评论痛点提取 Skill。
- Shopify 导入包生成 Tool。
- Google Trends 查询 MCP。
- SKU 图片生成 Skill。
- Listing 合规检查 Agent。
- 爆品筛选 Team。
- 新品上架 Workflow Skeleton。
- 广告投放建议 Workflow。

当数量变多后，会出现：

| 问题 | 例子 |
| --- | --- |
| 名字相似 | `Listing 优化 Skill`、`SEO Listing Skill`、`Shopify Listing Skill` |
| 能力重复 | 两个 Skill 都能做竞品卖点提炼 |
| 边界不清 | 一个 Tool 能生成 Shopify payload，但不能写 Shopify |
| 版本混乱 | v1 成本低但效果差，v3 效果好但慢 |
| 场景错配 | Amazon 选品 Skill 被拿去做 Shopify 商品写入 |
| 权限错配 | 某 MCP 需要 Amazon 数据源，但当前租户未授权 |
| 质量不稳定 | 有些能力是探索沉淀的草稿，还没通过测试 |
| 过期 | 外部平台 API 变了，旧 connector 不能用了 |

所以必须把“能力发现和治理”当作平台核心模块。

## Capability Registry

### Capability 是统一抽象

无论是 Skill、Tool、MCP、Agent、Team、Workflow Skeleton、Workflow，都应该进入统一能力目录。

```text
Capability
- skill
- tool
- mcp_connector
- agent
- team
- workflow_skeleton
- workflow
```

Orchestrator 先查 Capability Registry，再决定调用什么。

### CapabilityCard

每个能力必须有一张能力卡片。

```text
capability_cards
- id
- capability_type
- capability_id
- name
- short_description
- long_description
- ecommerce_domain
- supported_tasks
- input_requirements
- output_contract
- suitable_when
- not_suitable_when
- required_permissions
- required_data_sources
- risk_level
- cost_level
- latency_level
- model_requirements
- examples
- counter_examples
- quality_score
- success_rate
- last_used_at
- owner
- status = draft / testing / active / deprecated / disabled
```

## Skill 必须有能力边界说明

Skill 不能只是一段 prompt 或说明文字。

每个 Skill 至少要写清楚：

```text
SkillSpec
- name
- purpose
- task_types
- input_context_needed
- output_format
- works_well_when
- does_not_work_when
- required_tools
- required_knowledge
- risk_notes
- examples
- evaluation_cases
- version
- owner
- status
```

例子：

```text
Skill: 竞品卖点提炼 Skill

能力：
- 从竞品页面、评论摘要、价格和图片描述中提炼可借鉴卖点。
- 将竞品卖点改写为自有品牌表达。

适用：
- 用户有竞品链接。
- 需要生成 ProductDraft、ListingBrief、ImageBrief。

不适用：
- 不能直接复制竞品文案。
- 不能判断商标/专利法律结论，只能提示风险。
- 不能写入 Shopify。

输入：
- competitor_snapshots
- target_market
- brand_rules
- product_category

输出：
- differentiated_selling_points
- risk_notes
- source_references
```

## MCP / Tool 也必须有边界说明

MCP / Tool 比 Skill 风险更高，因为它可能连接外部系统。

每个 MCP / Tool 必须有：

```text
ToolSpec / ConnectorSpec
- tool_key
- provider
- operation_type = read / write / generate / transform / external_action
- risk_level
- required_scopes
- input_schema
- output_schema
- side_effects
- rate_limits
- cost_policy
- auth_provider = nango / platform_key / tenant_key
- allowed_callers
- approval_required
- dry_run_supported
- audit_level
- examples
- failure_modes
```

例子：

```text
Tool: shopify.product.create

能力：
- 在 Shopify 中创建商品。

边界：
- 只能通过 Tool Gateway 调用。
- 必须有 Shopify write_product scope。
- 必须经过 Approval。
- 必须支持 dry-run。
- 不能由 Orchestrator 直接调用。

输入：
- ShopifyImportPackage

输出：
- created_product_id
- media_upload_status
- metafield_status
```

## 系统如何自动找到已有能力

### 不能只靠向量检索

能力发现不能只做 embedding similarity。

原因：

- 权限不匹配的能力不能返回。
- draft / disabled 能力不能随便用。
- 高风险写工具不能被普通对话直接选中。
- 相似名字不代表适合当前任务。

推荐流程：

```text
1. Intent Understanding
2. Capability Query 构造
3. Hard Filter
4. Semantic Retrieval
5. Rule Ranking
6. LLM Rerank
7. Compatibility Check
8. PlanDraft 引用能力
9. 用户确认
```

### Step 1: Intent Understanding

Orchestrator 先把用户对话转成结构化查询：

```text
CapabilityQuery
- task_goal
- ecommerce_domain
- required_inputs
- expected_outputs
- data_sources
- risk_level
- target_systems
- tenant_id
- shop_id
- user_role
- constraints
```

### Step 2: Hard Filter

先过滤掉一定不能用的能力：

- 当前租户不可见。
- 用户无权限。
- 数据源未授权。
- 状态不是 active/testing。
- 风险等级超过当前流程允许。
- 需要的 provider 不可用。
- 成本超过预算。

### Step 3: Semantic Retrieval

再用语义检索找相似能力：

- capability name。
- short description。
- long description。
- task examples。
- suitable_when。
- not_suitable_when。
- input/output schema 描述。

### Step 4: Rule Ranking

用确定性规则加权：

- task_type 匹配。
- domain 匹配。
- input schema 匹配。
- output contract 匹配。
- required tool 可用。
- success_rate。
- quality_score。
- 最近是否被成功使用。
- 是否被当前 tenant 收藏或启用。

### Step 5: LLM Rerank

LLM 只在候选集里做解释型排序：

```text
候选 A 为什么适合。
候选 B 为什么不适合。
是否需要组合多个能力。
是否缺少新 Tool。
是否应该进入探索模式。
```

LLM 不能越过 Hard Filter。

### Step 6: Compatibility Check

最终再校验：

- 输入是否够。
- 输出是否能接下游。
- 工具是否可执行。
- 审批是否齐全。
- 风险是否可接受。

## 能力发现的输出

Orchestrator 不应该直接说“我找到了一个工具然后用它”。

应该生成：

```text
CapabilitySelection
- selected_capabilities
- why_selected
- not_selected_candidates
- missing_capabilities
- required_user_confirmation
- risk_notes
```

用户可见表达：

```text
我找到三个可复用能力：
1. 竞品卖点提炼 Skill：适合从竞品链接提炼差异化卖点。
2. Shopify 导入包生成 Tool：适合把结构化商品资料转成 Shopify payload，但不会直接写入。
3. Listing 上架审核 Team：适合在写入前做 SEO、文案、合规检查。

还缺一个能力：自动生成 SKU 场景图的图片资产工作流。目前可以先用通用图片生成流程探索一次，成功后再沉淀为 Skill 或 Workflow Skeleton。
```

## 能力过多时的治理

### 1. 去重和合并

系统应该定期发现：

- 名称相似。
- 输入输出相同。
- 任务场景相同。
- 成功率差异明显。

输出：

```text
CapabilityMergeSuggestion
- primary_capability
- duplicate_candidates
- reason
- migration_plan
```

### 2. 质量评分

每个能力要有质量指标：

```text
quality_score
- schema_success_rate
- task_success_rate
- approval_pass_rate
- user_edit_rate
- user_reuse_rate
- cost_score
- latency_score
- failure_rate
- complaint_rate
```

### 3. 生命周期

```text
draft -> testing -> active -> deprecated -> disabled -> archived
```

不是所有沉淀能力都能 active。

### 4. 场景覆盖图

平台需要知道哪些场景已有能力、哪些没有：

```text
CapabilityCoverage
- listing
- product_research
- competitor_analysis
- image_generation
- shopify_import
- ads
- customer_service
- inventory
```

这能帮助平台决定下一步该建设什么。

## 前端应该怎么展示

### 能力搜索页

用户可以搜索：

```text
竞品链接生成 Shopify 上架资料
```

前端展示：

- 推荐能力。
- 能力类型。
- 适用场景。
- 边界说明。
- 需要权限。
- 成功率。
- 最近使用。
- 示例。

### 对话中的能力引用

AI 在对话中应该透明说明：

```text
我会复用以下能力：
- 竞品卖点提炼 Skill
- Shopify 导入包生成 Tool
- Listing 上架审核 Team

这次不会直接复用完整 Workflow，因为你的需求包含图片生成和 A+ 页面设计，和已有流程不完全一致。我会用通用上架 Workflow Skeleton 生成一个临时计划。
```

### 管理后台

管理员可以：

- 查看所有能力。
- 查看能力边界。
- 启用/禁用能力。
- 合并重复能力。
- 审核 SkillDraft / ToolRequirementDraft。
- 查看能力质量指标。
- 看哪些能力被哪些 Agent/Team/Workflow 引用。

## 对现有架构的新增要求

### 1. Business Registry 升级为 Capability Registry

原来的业务 Registry 管 Agent、Team、Workflow、Tool、Model。

现在要明确升级为：

```text
Capability Registry
- Agent
- Team
- Workflow
- WorkflowSkeleton
- Skill
- Tool
- MCP / Connector
- ModelPolicy
- MemoryPolicy
```

### 2. Orchestrator 必须使用 Capability Retrieval

Orchestrator 不能只靠 prompt 记忆能力。

必须调用：

```text
search_capabilities
get_capability_card
check_capability_compatibility
propose_capability_selection
```

### 3. Skill / MCP 不能没有说明

没有说明、没有边界、没有输入输出、没有风险说明的能力，不能进入 active。

### 4. 能力版本也要治理

Skill、Tool、MCP、Workflow Skeleton 都要有版本：

```text
skill_versions
tool_versions
mcp_connector_versions
workflow_skeleton_versions
```

每次运行要记录使用了哪个版本。

## 建议新增技术验证

| 文档 | 目的 |
| --- | --- |
| `CapabilityRegistry能力目录与检索验证.md` | 验证统一能力目录、能力卡片、过滤、检索、rerank |
| `SkillSpec能力边界说明验证.md` | 验证 Skill 必须有适用场景、不适用场景、输入输出、示例和评估 |
| `MCP与ToolSpec边界说明验证.md` | 验证外部工具/MCP 的 scope、side effects、approval、dry-run、audit |
| `能力质量评分与去重治理验证.md` | 验证能力质量评分、重复能力发现、合并和下线 |
| `Orchestrator能力发现与选择验证.md` | 验证 AI 如何从大量能力中找出合适能力并解释选择原因 |

## 最终结论

如果 Skill、MCP、Tool 等能力沉淀太多，系统不能靠人记，也不能靠 prompt 硬塞。

必须把能力资产化：

```text
每个能力都有卡片。
每个能力都有边界。
每个能力都有版本。
每个能力都有质量指标。
每个能力都能被检索、过滤、排序、解释和审计。
```

AI 可以自己找到已有能力，但前提是：

```text
Capability Registry 提供结构化能力目录。
Capability Retrieval 做硬过滤 + 语义检索 + 规则排序 + LLM rerank。
Orchestrator 只从合法候选里选择能力。
用户能看到 AI 为什么选这些能力。
```

这会避免平台后期变成“能力仓库很大，但真正用的时候找不到、选不准、边界不清”的状态。
