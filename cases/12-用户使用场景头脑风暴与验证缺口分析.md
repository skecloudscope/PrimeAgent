# 12 用户使用场景头脑风暴与验证缺口分析

## 文档目的

前面 `00-11` 的文档是从技术模块出发，验证 Agent、Team、Workflow、Registry、Orchestrator、Tool Gateway、Approval、LLM Provider 是否可行。

本文件反过来从用户真实使用出发，先列出跨境电商用户可能怎么用这个平台，再反推当前 `agent协同验证` 目录是否有遗漏、哪些技术点还需要新增验证。

核心问题：

```text
用户真实会怎么提需求？
系统应该路由到哪个 Agent / Team / Workflow？
哪些场景会突破当前 Listing MVP？
哪些能力现在还没有被充分验证？
```

## 当前 00-11 总结

### 00 Agent 协同验证总览

当前结论：

- Agent 协同验证是平台核心，不是普通子任务。
- 主链路是：用户自然语言 -> Orchestrator -> Workflow -> Agent -> Team -> Tool Gateway -> Approval -> Audit -> Memory Candidate。
- 第一阶段先证明 `Listing 优化并写回 Shopify` 可跑通。

我对它的判断：

- 方向正确。
- 现在偏“技术模块总览”，缺少“用户真实任务地图”。
- 本文件就是补这一层。

### 01 Agent 创建与配置验证

当前结论：

- Agno Agent 可以满足第一版专业 Agent 的创建和运行。
- 业务平台必须自建 AgentTemplate、AgentInstance、AgentVersion、ModelPolicy、ToolPermissionPolicy、KnowledgeScope、MemoryPolicy。
- `Listing 优化 Agent` 第一版只生成建议，不直接写 Shopify。

我对它的判断：

- 正确。
- 但后续还要补充“Agent 输入上下文包”的统一规范，例如商品、店铺、市场、语言、品牌、竞品、历史表现等如何组合给 Agent。

### 02 Agent 版本迭代与优化验证

当前结论：

- 第一版不做自动优化。
- 采用受控闭环：记录证据 -> 形成优化案例 -> 生成人工可审优化建议 -> 创建 draft 版本 -> 测试集回放 -> 人工发布 -> 可回滚。
- Agno trace、metrics、eval 可以作为底层证据，但商业优化系统必须自建。

我对它的判断：

- 正确，而且这是平台长期壁垒。
- 但还缺少“用户反馈类型”的业务分类，例如用户手动改了标题、拒绝审批、商品发布后转化改善/变差，这些反馈如何进入优化案例。

### 03 Agent 管理审核与发布验证

当前结论：

- Agno 的 component/config/draft/published/current_version 能借鉴。
- 商业 Agent 生命周期、审核、发布门禁、运行准入必须自建。
- write tool 权限变更必须重新审核。

我对它的判断：

- 正确。
- 但还需要明确“谁能创建 Agent、谁能编辑 Agent、谁能发布 Agent”的角色模型，尤其是平台官方 Agent、租户自定义 Agent、租户员工临时 Agent 的差异。

### 04 Team 调用 Agent 与协同模式验证

当前结论：

- Agno Team 支持 `route / coordinate / broadcast / tasks`。
- MVP 最推荐 `broadcast`，用于 Listing 多专家评审。
- Team 不做主流程控制，不直接执行 write tool。

我对它的判断：

- 正确。
- 但需要补充“Team 评审冲突处理”，例如 SEO Agent 建议加关键词，合规 Agent 认为关键词有风险，最终谁裁决。

### 05 Team 版本迭代与成员治理验证

当前结论：

- TeamVersion 必须锁定成员 AgentVersion。
- 成员 Agent 升级不自动影响已发布 Team。
- Team 更新必须走 testing 和审核。

我对它的判断：

- 正确。
- 但还需要验证“团队成员替换/禁用后的影响分析”，例如 Compliance Agent 因效果差被暂停，哪些 Team/Workflow 受影响。

### 06 Workflow 整体编排逻辑验证

当前结论：

- Agno Workflow 可以作为 MVP 主编排层。
- Workflow 能串起 function step、Agent step、Team step、审批、错误处理、暂停恢复。
- Tool Gateway 是唯一能读写 Shopify 的地方。

我对它的判断：

- 正确。
- 但当前主要围绕单个商品 Listing。后续需要补充批量任务、长任务、部分成功、批次审批、任务取消、任务恢复。

### 07 Workflow 版本迭代与回放验证

当前结论：

- WorkflowVersion 是商业流程版本。
- 历史 run 不跟随版本升级。
- 回放不允许真实写 Shopify。
- rollback 只切 active_version_id。

我对它的判断：

- 正确。
- 但还需要验证“线上运行中的 Workflow 遇到版本发布怎么办”，例如一个长任务运行到一半时 active version 切换，必须保证当前 run 继续使用旧版本。

### 08 LLM Provider 与模型路由验证

当前结论：

- Agno 支持多 provider、fallback、token/cost metrics。
- 平台管理 provider key。
- 租户选择模型档位，业务层控制模型权限和预算。

我对它的判断：

- 正确。
- 但还需要补充“模型输出质量和商业指标的关系”，例如便宜模型导致 Listing 质量下降，如何通过评估和审批发现。

### 09 Agent Registry 与组件发现验证

当前结论：

- Agno Registry 是运行时对象注册表，不是产品级 Registry。
- 业务 Registry 必须自建。
- 业务 Registry 负责租户、权限、版本、审核、行业场景、工具策略、模型策略、记忆策略和 Runtime Plan。

我对它的判断：

- 正确，是整个商业平台最关键的事实源。
- 但还需要补充“用户看见的入口目录”，例如首页是按 Agent 展示，按 Workflow 展示，还是按业务问题展示。

### 10 Orchestrator 总控 Agent 验证

当前结论：

- 需要 Orchestrator。
- Orchestrator 是入口和路由层，不是执行层。
- 它只能启动白名单 Workflow，不能直接写 Shopify。
- Workflow 内部路由交给 Router / Condition。

我对它的判断：

- 正确。
- 但还需要验证“多轮任务上下文”，例如用户连续说“再把第二个也优化一下”“这个不要写回，先保存草稿”，Orchestrator 如何指向正确的前一个 run / product / approval。

### 11 Agent 协同端到端原型验证

当前结论：

- 第一条端到端原型固定为 `Listing 优化并写回 Shopify`。
- Shopify/Nango/Clerk 可以先 mock。
- 必须真实验证 Agno Agent、Team、Workflow、Runtime Plan、版本绑定、Approval、Tool Gateway、Audit Log。

我对它的判断：

- 正确，已经具备开始写原型的形状。
- 但从用户视角看，单商品 Listing 只是第一类场景。需要提前列出第二批和第三批场景，避免 MVP 表设计过窄。

## 用户角色头脑风暴

### 店铺老板 / 创始人

典型诉求：

- 想看店铺哪里有问题。
- 想快速提升销量。
- 想减少运营人员重复劳动。
- 不一定知道应该找哪个 Agent。

系统入口：

- Orchestrator。
- 店铺诊断 Workflow。
- 周报/月报 Workflow。
- 高风险操作审批。

核心关注：

- 结果是否可信。
- 会不会乱改店铺。
- 成本和时间是否可控。

### 运营负责人

典型诉求：

- 批量优化商品。
- 检查 Listing 质量。
- 跟踪优化后效果。
- 安排团队成员审核和执行。

系统入口：

- Listing 优化 Workflow。
- 批量 Listing Review Workflow。
- 任务队列。
- 审批列表。

核心关注：

- 批量任务是否可控。
- 哪些商品成功、哪些失败。
- Agent 建议能否符合品牌调性。

### Listing 运营

典型诉求：

- 写标题、五点、描述、SEO。
- 根据竞品优化关键词。
- 多语言本地化。
- 检查合规词。

系统入口：

- Listing 优化 Agent。
- SEO Review Agent。
- Compliance Review Agent。
- Listing Review Team。

核心关注：

- 输出是否可编辑。
- 能否保存草稿。
- 能否对比新旧版本。

### 广告投手

典型诉求：

- 看广告花费是否异常。
- 找出低 ROAS 广告。
- 生成调价或停投建议。
- 希望自动执行预算调整，但风险很高。

系统入口：

- Ads Diagnosis Workflow。
- Ads Optimization Agent。
- Ads Risk Review Team。
- Approval。

核心关注：

- 广告预算写操作必须强审批。
- 需要连接广告平台，后续可能是 Google Ads、Meta、TikTok Ads、Amazon Ads。

### 客服负责人

典型诉求：

- 总结差评原因。
- 生成客服回复草稿。
- 批量分类用户问题。
- 不希望 Agent 直接乱发消息。

系统入口：

- Review Analysis Workflow。
- Customer Service Draft Agent。
- Message Approval Workflow。

核心关注：

- 外发消息必须审批。
- 语气和品牌规范要稳定。
- 隐私和 PII 要控制。

### 供应链 / 库存负责人

典型诉求：

- 库存预警。
- 销量预测。
- 补货建议。
- 识别滞销品。

系统入口：

- Inventory Diagnosis Workflow。
- Replenishment Suggestion Agent。
- Forecast Review Team。

核心关注：

- 建议可以自动生成，但下采购单或改库存必须强审批。
- 数据来源可能来自 Shopify、ERP、WMS、Google Sheets。

### 管理员 / 平台配置人员

典型诉求：

- 连接 Shopify。
- 配置哪些员工能用哪些 Agent。
- 审核 Agent 版本。
- 查看运行日志和成本。
- 禁用危险 Workflow。

系统入口：

- 管理后台。
- Registry 管理。
- Approval Policy。
- Tool Policy。
- Model Policy。
- Audit Log。

核心关注：

- 权限边界。
- 审计追溯。
- 成本预算。

## 用户使用场景例子

### 场景 1：单个商品 Listing 优化并写回

用户说：

```text
帮我优化这个商品的 Listing，并准备写回 Shopify。
```

系统路径：

```text
Orchestrator
  -> 识别 listing_optimize_writeback
  -> Registry 解析可用 Workflow
  -> Listing Workflow
  -> Shopify read
  -> Listing Optimization Agent
  -> Listing Review Team
  -> ListingDiff
  -> Approval
  -> Tool Gateway write
```

当前覆盖情况：

- 已覆盖，是 11 的主原型。

还需要注意：

- 用户可能只想保存草稿，不想写回。
- 用户可能想只改标题，不改描述。
- 用户可能需要中英文/多市场版本。

### 场景 2：批量优化 50 个商品

用户说：

```text
把这 50 个商品都优化一下，先生成草稿，不要直接写回。
```

系统路径：

```text
Orchestrator
  -> 识别 batch_listing_optimize
  -> 收集商品范围
  -> Batch Workflow
  -> 为每个商品生成子任务
  -> 每个子任务调用 Listing Agent / Review Team
  -> 汇总结果
  -> 用户批量筛选
```

当前覆盖情况：

- 部分覆盖。Workflow 单商品链路覆盖了，但批量任务、部分成功、批量审批没有覆盖。

建议新增验证：

- 批量 Workflow 与子任务编排验证。
- 批量运行状态、部分失败和重试验证。
- 批量审批与逐项审批验证。

### 场景 3：只做 Listing 质量体检

用户说：

```text
帮我检查一下这些商品的 Listing 哪些质量差。
```

系统路径：

```text
Listing Audit Workflow
  -> 读取商品列表
  -> Listing Quality Agent
  -> SEO / Copy / Compliance Team
  -> 输出评分和问题列表
```

当前覆盖情况：

- Team 评审能力覆盖了。
- 但“只读诊断、不生成写回计划”的 Workflow 没有单独定义。

建议新增验证：

- 只读诊断 Workflow。
- 评分 schema。
- 诊断结果如何进入后续优化 Workflow。

### 场景 4：用户根据效果反馈优化 Agent

用户说：

```text
这个 Agent 上次写的标题太营销化了，不符合我们品牌，以后收敛一点。
```

系统路径：

```text
Feedback Capture
  -> AgentFeedback
  -> OptimizationCase
  -> OptimizationSuggestion
  -> AgentVersion draft
  -> 测试集回放
  -> 人工审核发布
```

当前覆盖情况：

- 02 覆盖了迭代闭环。
- 但缺少反馈分类、反馈证据、反馈作用范围。

建议新增验证：

- 用户反馈与 Agent 优化案例验证。
- 反馈是只影响当前商品、当前店铺、当前租户，还是平台模板。

### 场景 5：审批时用户修改 Agent 生成的写回内容

用户操作：

```text
审批页里用户不完全接受 Agent 的标题，手动改了两个词后批准写回。
```

系统路径：

```text
Approval UI
  -> 用户编辑 ListingDiff
  -> 保存 EditedDiff
  -> Tool Gateway 按 EditedDiff 写回
  -> AuditLog 记录 original_diff 和 final_diff
  -> Feedback 进入优化案例
```

当前覆盖情况：

- 审批和写回覆盖了。
- 但“审批时编辑 diff”和“编辑痕迹进入优化”还没有充分覆盖。

建议新增验证：

- Approval 编辑 diff 验证。
- EditedDiff 与 Agent 原始输出的对比记录。

### 场景 6：用户拒绝审批并要求重做

用户操作：

```text
这个优化不行，标题太长，重新生成一个更简洁的。
```

系统路径：

```text
Approval rejected
  -> reject_reason
  -> Workflow revise path
  -> Listing Agent rerun with reject_reason
  -> Review Team rerun
  -> New ApprovalRequest
```

当前覆盖情况：

- 06/07 提到 reject/retry。
- 但没有作为单独用户场景详细验证。

建议新增验证：

- 审批拒绝后的 revise workflow。
- 一个 workflow_run 是否生成多个 approval_request。
- 新旧 diff 的版本关系。

### 场景 7：店铺诊断

用户说：

```text
帮我看看店铺最近 30 天哪里有问题。
```

系统路径：

```text
Orchestrator
  -> 识别 store_diagnosis
  -> 收集时间范围
  -> Store Diagnosis Workflow
  -> 读取商品、订单、流量、转化、广告数据
  -> Diagnosis Team
  -> 输出问题优先级和建议任务
```

当前覆盖情况：

- 10 提到这个场景。
- 但没有 Workflow、数据 schema、Team 结构、数据源权限验证。

建议新增验证：

- 店铺诊断 Workflow 验证。
- 多数据源读取和权限 scope 验证。
- 诊断结果转任务验证。

### 场景 8：广告预算建议

用户说：

```text
找出最近浪费预算的广告，并给我调整建议。
```

系统路径：

```text
Ads Diagnosis Workflow
  -> 读取广告数据
  -> Ads Analysis Agent
  -> Profit / Risk Review Team
  -> 输出预算调整建议
  -> 如果要执行，进入强审批
```

当前覆盖情况：

- 当前文档几乎没有覆盖广告域。

建议新增验证：

- 高风险数值型写操作验证。
- 广告预算 Tool Gateway policy。
- 预算变更 diff 和审批。

### 场景 9：客服回复草稿

用户说：

```text
帮我回复这些差评，语气诚恳一点，不要直接发送。
```

系统路径：

```text
Review Reply Workflow
  -> 读取评论/工单
  -> Customer Reply Agent
  -> Brand Voice Review Agent
  -> 输出回复草稿
  -> 人工批准后发送
```

当前覆盖情况：

- Agent/Workflow/Approval 能复用。
- 但外发消息类工具、PII、品牌语气、批量草稿没有覆盖。

建议新增验证：

- 外发消息审批验证。
- PII 检测和脱敏验证。
- 品牌语气 Memory/Knowledge 验证。

### 场景 10：库存和补货建议

用户说：

```text
哪些 SKU 快断货了？给我一个补货建议。
```

系统路径：

```text
Inventory Diagnosis Workflow
  -> 读取库存、销量、在途、交期
  -> Forecast Agent
  -> Replenishment Agent
  -> 输出补货建议
  -> 下采购单必须审批
```

当前覆盖情况：

- 当前文档没有覆盖供应链/库存域。

建议新增验证：

- 多源数据 join 验证。
- 数值预测 Agent 输出验证。
- 建议和真实执行动作分离验证。

### 场景 11：连接 Shopify 失败或 scope 不足

用户说：

```text
帮我读取这个商品。
```

系统发现：

```text
Nango connection missing / expired / missing scope
```

系统路径：

```text
Orchestrator
  -> 检查 connection_status
  -> 引导用户授权
  -> 授权成功后继续 Workflow
```

当前覆盖情况：

- Nango 边界已讲清楚。
- 但 connection 异常恢复、授权后 resume 没有验证。

建议新增验证：

- Connector 状态与 Workflow pause/resume 验证。
- missing scope 的用户提示和重授权验证。

### 场景 12：用户正在运行长任务时切换店铺

用户操作：

```text
用户在 A 店铺发起批量优化，然后切到 B 店铺继续聊天。
```

系统风险：

- Orchestrator 上下文错乱。
- Workflow run 绑定错 shop。
- Approval 显示错店铺。

当前覆盖情况：

- 现有文档强调 tenant/shop 绑定。
- 但多店铺上下文切换还没有单独验证。

建议新增验证：

- 会话上下文与运行上下文隔离验证。
- Approval 必须显示 tenant/shop/product 明细。

### 场景 13：员工权限不足

用户说：

```text
把这批商品直接写回 Shopify。
```

系统发现：

```text
user role = listing_editor
没有 write approval 权限
```

系统路径：

```text
允许生成草稿
禁止批准写回
请求 owner 审批
```

当前覆盖情况：

- 03/09/10 提到权限。
- 但角色矩阵和委托审批没有展开。

建议新增验证：

- RBAC 与 Approval 权限验证。
- 发起人和审批人不能同一人的策略验证。

### 场景 14：Agent 输出格式错误

系统情况：

```text
Listing Agent 没有输出合法 ListingOptimizationDraft。
```

系统路径：

```text
parser retry
  -> fallback model
  -> fail-close
  -> 标记 step failed
  -> 不进入写回
```

当前覆盖情况：

- 01/04/06/08 都提到 structured output。
- 但 parser retry、fallback、最终失败后的用户体验还需要验证。

建议新增验证：

- 结构化输出失败处理验证。
- output schema migration 验证。

### 场景 15：用户想创建自己的 Agent

用户说：


当前 11 的端到端原型是单商品。

真实跨境电商场景里，批量任务非常常见：

- 批量优化商品。
- 批量生成回复草稿。
- 批量检查 Listing 质量。
- 批量广告诊断。

- 部分失败。
- 批量重试。
- 批量审批。
- 批量成本上限。
- 批量 replay 禁止写操作。

### 遗漏 2：Approval 编辑、拒绝、重做闭环

当前文档确认写回前必须审批，但还不够细。

真实用户会：

- 直接批准。
- 编辑后批准。
- 拒绝。
- 要求重做。
- 转交给老板审批。

要验证：

- final_diff。

- revise path。

- approver 权限。
- 发起人和审批人是否可同一人。

### 遗漏 3：用户反馈如何进入 Agent/Team/Workflow 优化

真实反馈来源更多：

## 对当前总结是否有问题的判断

当前 `00-11` 的总结没有方向性问题。

它最强的地方是：

- 清楚地区分了 Agno runtime 和业务平台控制层。
- 把 Workflow 定为商业流程主控，而不是让 Agent 自由执行。
- 把 Tool Gateway 定为外部写操作边界。
- 把 Approval 放在写操作之前。
- 把 Registry 定为业务事实源。
- 把 Orchestrator 限定为入口和路由层。

当前不足不是“错”，而是“还偏第一条 Listing 单商品链路”。

接下来需要补的不是推翻，而是扩大验证半径：

```text
单商品 -> 批量任务
直接批准 -> 编辑/拒绝/重做
单店铺 -> 多店铺上下文
正常连接 -> 连接异常和授权恢复
写回流程 -> 只读诊断流程
平台预设 Agent -> 用户自定义 Agent
后端运行 -> 前端执行体验
Memory Candidate 概念 -> 可控记忆写入治理
```
