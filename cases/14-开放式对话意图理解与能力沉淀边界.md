# 14 开放式对话意图理解与能力沉淀边界

## 文档目的

本文件记录一个关键产品判断：

跨境电商场景里的用户对话大概率是开放的、渐进的、不完全结构化的。用户有时会一次性把所有需求说出来，有时会一步步补充，有时自己也不知道最终应该执行什么流程。

因此平台不能假设大部分业务都能预先固化成稳定 Workflow。

更合理的设计是：

```text
AI 先理解意图
不理解就追问
理解后生成可确认的 PlanDraft
执行过程中持续吸收用户补充
执行完成后判断应该沉淀什么
```

最终更稳定可复用的通常不是完整 Workflow，而是：

- Skill。
- Tool。
- MCP / Connector 能力。
- Agent 能力。
- 简单 Team 统筹能力。
- 通用 Workflow 骨架。
- 领域知识和操作偏好。

完整 Workflow 可以沉淀，但不能假设下一次一定能原样复用。

## 核心判断

### 1. 用户意图经常不是一次性完整表达

用户可能这样说：

```text
帮我看看这个品能不能做。
```

然后逐步补充：

```text
主要看美国市场。
顺便看一下 Amazon 上有没有竞品。
如果可以做，帮我生成 Shopify 上架资料。
图片也要做。
先不要写回店铺。
```

这不是固定 Workflow，而是一个逐步展开的业务探索过程。

系统必须支持：

- 意图识别。
- 缺口追问。
- 上下文保持。
- 用户临时改目标。
- 用户追加约束。
- 用户随时暂停、改方向、收敛结果。

### 2. Orchestrator 不只是 Workflow 选择器

之前我们把 Orchestrator 定位为：

```text
识别意图 -> 选择白名单 Workflow -> 启动 Workflow
```

这个定位对稳定流程成立，但对开放式对话不够。

新的定位应该是：

```text
Orchestrator = 对话理解 + 需求澄清 + 动态计划 + 能力路由 + 执行监督 + 沉淀判断
```

它仍然不能直接执行高风险动作，但它要能处理“没有现成 Workflow”的情况。

### 3. Workflow 不一定是最终沉淀物

一次复杂任务跑完后，系统要判断沉淀哪类能力。

| 沉淀物 | 适合情况 | 例子 |
| --- | --- | --- |
| Skill | 做法稳定，但每次步骤会变 | “如何从竞品链接提炼卖点并改写成自有品牌表达” |
| Tool | 某个确定性动作频繁出现 | “把 ProductDraft 转成 Shopify import payload” |
| MCP / Connector | 需要接入新外部系统或新数据源 | “查询 Amazon 商品和评论摘要” |
| AgentVersion | 某个专业角色变稳定 | “宠物用品 Listing 优化 Agent” |
| TeamVersion | 多个专家组合稳定 | “Listing 上架审核 Team” |
| Workflow Skeleton | 大框架稳定，小步骤动态 | “外部研究 -> 结构化 -> 生成资产 -> 审核 -> 导入” |
| Full Workflow | 步骤、输入、工具、审批都高度稳定 | “单商品 Listing 优化并写回 Shopify” |

所以平台的沉淀策略应该是：

```text
默认沉淀 Skill / Tool / Team / Workflow Skeleton
只有高频、稳定、可测试的流程才沉淀为 Full Workflow
```

## 对话模式设计

### 模式 A：用户一次性说完整需求

用户：

```text
从这三个竞对链接抓数据，提炼卖点，生成 Shopify 标题、五点和详情页，再做 5 张场景图和 3 张 SKU 图，最后整理成 Shopify 导入格式。先不要写回。
```

系统应该：

```text
1. 识别这是 new_product_launch_asset_pipeline。
2. 抽取已知参数：竞对链接、目标输出、是否写回。
3. 检查缺失参数：目标市场、品牌规则、店铺、产品类目、图片风格。
4. 生成 PlanDraft。
5. 请求用户确认计划。
6. 执行只读和生成类步骤。
7. 输出资产包和 Shopify import package。
8. 问用户是否保存为后续可复用能力。
```

### 模式 B：用户一步步说需求

用户：

```text
帮我看看这个品。
```

系统应该追问：

```text
你想从哪个角度看？选品机会、竞品分析、上架资料、广告投放，还是利润风险？
```

用户：

```text
先看选品机会，主要看美国。
```

系统：

```text
1. 创建 ConversationGoal。
2. 进入 ProductResearchExploration。
3. 追问必要参数。
4. 执行外部研究。
5. 根据结果建议下一步：上架资料、广告测试、放弃、继续找供应链。
```

### 模式 C：用户中途改变目标

用户：

```text
先不要生成 Shopify 数据了，我想先看看这个品有没有侵权风险。
```

系统应该：

```text
1. 暂停当前 PlanDraft 的后续步骤。
2. 插入 Compliance Research step。
3. 重新评估风险。
4. 如果风险高，建议停止上架流程。
5. 保留已生成的中间结果，但不继续写入 Shopify。
```

### 模式 D：用户要求复用上次做法

用户：

```text
上次那个选品分析做得不错，以后类似品类就按那个逻辑来。
```

系统不应该直接创建 full Workflow。

系统应该先生成：

```text
SkillDraft:
- 选品分析指标。
- 数据源优先级。
- 过滤逻辑。
- 输出报告结构。
- 风险检查点。

WorkflowSkeletonDraft:
- 收集产品信息。
- 外部研究。
- 内部数据对比。
- 机会评分。
- 输出报告。
```

如果后续多次运行稳定，再升级为 Full Workflow。

## 新增核心对象

### ConversationGoal

```text
conversation_goals
- id
- tenant_id
- user_id
- conversation_id
- current_goal
- status = discovering / planning / executing / paused / completed / abandoned
- confidence
- missing_information
- constraints
- active_plan_draft_id
- created_at
- updated_at
```

用途：

- 保存这轮对话到底在做什么。
- 支持用户一步步补充需求。
- 防止 Orchestrator 每轮都重新理解。

### ClarificationQuestion

```text
clarification_questions
- id
- conversation_goal_id
- question
- reason
- required = true / false
- answered_by_user
- answer
- created_at
```

用途：

- 记录 AI 为什么追问。
- 区分必须信息和可选信息。
- 后续可优化追问策略。

### PlanDraft

```text
plan_drafts
- id
- tenant_id
- user_id
- conversation_goal_id
- goal
- proposed_steps
- known_inputs
- missing_inputs
- assumptions
- required_tools
- estimated_cost
- risk_level
- approval_points
- status = draft / waiting_user_confirmation / confirmed / executing / completed / cancelled
```

用途：

- 对开放式任务先生成计划。
- 让用户确认。
- 不等于 Workflow。

### ExplorationRun

```text
exploration_runs
- id
- tenant_id
- user_id
- plan_draft_id
- status
- run_graph_id
- started_at
- completed_at
- final_outputs
- user_satisfaction_signal
```

用途：

- 执行一次非固定流程。
- 记录动态步骤。
- 为能力沉淀提供证据。

### CapabilityCandidate

```text
capability_candidates
- id
- tenant_id
- source_type = exploration_run / workflow_run / user_feedback
- source_id
- candidate_type = skill / tool / mcp / agent / team / workflow_skeleton / workflow
- name
- reason
- evidence
- draft_payload
- status = proposed / accepted / rejected / converted
```

用途：

- 统一管理“这次执行后值得沉淀什么”。
- 不急着全部变成 Workflow。

## 能力沉淀判断规则

### 什么时候沉淀 Skill

适合：

- 做法稳定。
- 但每次输入不同、步骤顺序可能变。
- 用户经常说“以后按这个判断逻辑”。

例子：

```text
选品评分方法。
竞品卖点提炼方法。
品牌语气改写方法。
广告投放前风险检查方法。
```

### 什么时候沉淀 Tool

适合：

- 动作确定。
- 输入输出结构稳定。
- 不需要 LLM 自由判断。

例子：

```text
ProductDraft -> ShopifyProductPayload。
ListingDiff -> Shopify update payload。
多站点商品表现查询。
图片 URL 批量下载和存储。
```

### 什么时候沉淀 MCP / Connector

适合：

- 需要连接新外部系统。
- 现有 Tool Gateway 没有这个能力。
- 多个 case 都反复需要。

例子：

```text
Amazon 商品搜索。
Amazon 评论摘要。
Google Trends 查询。
广告平台数据读取。
ERP 库存查询。
```

### 什么时候沉淀 Team

适合：

- 多个专家角色组合稳定。
- 但外部流程不固定。

例子：

```text
Product Opportunity Review Team
- Market Research Agent
- Profit Agent
- Compliance Agent
- Ads Agent
```

### 什么时候沉淀 Workflow Skeleton

适合：

- 大流程稳定。
- 具体步骤、工具、模型、数据源经常变。

例子：

```text
选品分析通用骨架：
1. 明确品类和市场
2. 外部需求研究
3. 竞品分析
4. 内部数据对比
5. 利润和风险评估
6. 输出机会报告
```

### 什么时候沉淀 Full Workflow

适合：

- 输入稳定。
- 步骤稳定。
- 工具稳定。
- 审批点稳定。
- 可以写测试集。
- 多次运行证明复用价值。

例子：

```text
单商品 Listing 优化并写回 Shopify。
批量 Listing 质量体检。
Shopify 商品导入包校验。
```

## 对现有架构的修正

### 1. Registry 不能只管 Workflow

业务 Registry 需要新增能力目录：

```text
Capability Registry
- skills
- tools
- mcp_connectors
- agents
- teams
- workflow_skeletons
- workflows
```

用户前端不应该只看到 Workflow 列表，而应该看到：

- 我能做哪些类型的事。
- 有哪些稳定能力可用。
- 哪些是探索模式。
- 哪些需要先确认计划。

### 2. Orchestrator 要能做意图澄清

Orchestrator 必须具备：

- 判断是否理解用户意图。
- 判断缺少哪些信息。
- 提出澄清问题。
- 用用户回答更新 ConversationGoal。
- 生成 PlanDraft。
- 判断该走现有 Workflow，还是进入 ExplorationRun。

### 3. Workflow 不再是唯一主路径

新的执行主路径有两条：

```text
稳定任务：
Conversation -> Orchestrator -> Published Workflow -> Run

开放任务：
Conversation -> Orchestrator -> PlanDraft -> ExplorationRun -> CapabilityCandidate
```

### 4. 前端必须支持计划确认

开放式任务不能直接跑。

前端需要展示：

- AI 理解的目标。
- 已知信息。
- 缺失信息。
- 假设。
- 计划步骤。
- 预计使用工具。
- 预计成本。
- 风险和审批点。

用户确认后才能执行。

### 5. 审计要记录“理解过程”

不仅要审计工具调用，还要审计：

- AI 如何理解用户意图。
- AI 追问了什么。
- 用户确认了什么计划。
- 哪些假设被采用。
- 中途目标如何变化。
- 最后沉淀了什么能力候选。

## 最终结论

这个平台不应该被设计成“Workflow 越多越好”。

更准确的产品形态是：

```text
对话是入口。
PlanDraft 是开放任务的临时计划。
ExplorationRun 是非固定流程的执行记录。
CapabilityCandidate 是经验沉淀入口。
Skill / Tool / MCP / Team / WorkflowSkeleton 是主要复用资产。
Full Workflow 只用于足够稳定、高频、可测试、可审批的流程。
```

这也意味着：

```text
AI 要先理解用户意图。
不理解就追问。
用户一次说全也能拆解。
用户逐步说也能持续收敛。
执行可以是动态的。
沉淀必须是受控的。
发布必须经过审核。
```

这个原则会让平台更适合跨境电商这种开放、复杂、经常变化的业务环境。
