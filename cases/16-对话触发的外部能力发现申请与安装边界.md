# 16 对话触发的外部能力发现申请与安装边界

## 文档目的

本文件修正一个重要边界：

外部 Skill、MCP、Tool、Agent 能力的发现和安装，不应该是系统后台无上下文地自动扫描、自动安装。

正确方式是：

```text
在对话中，用户提出一个当前平台能力无法完成的需求
  -> Orchestrator 判断出现能力缺口
  -> 去外部能力目录 / 已知 marketplace / 内部候选库里查找候选能力
  -> 向用户解释为什么需要这个能力
  -> 用户确认后申请安装或启用
  -> 管理员/策略审核
  -> 沙盒测试
  -> 才能进入租户可用能力
```

一句话：

```text
外部能力发现是 conversation-triggered，不是 background-autonomous。
```

## 典型场景

### 场景 1：用户要爬虫

用户说：

```text
帮我抓取这几个竞品页面，把标题、价格、图片、SKU 都整理出来。
```

系统发现：

```text
当前租户没有网页抓取能力。
任务需要浏览器自动化或网页抓取。
```

系统应该：

```text
1. 查询 Capability Registry。
2. 发现本地没有可用 crawler / browser tool。
3. 查询外部候选能力。
4. 找到 Playwright MCP / browser automation 能力。
5. 向用户说明：需要启用网页抓取能力，可能访问外部网页，会记录来源，不能绕过登录或付费墙。
6. 生成 CapabilityInstallRequest。
7. 等用户/管理员确认。
```

不能做：

```text
后台自己安装 Playwright MCP。
用户没确认就开始访问外部网页。
绕过网站限制或登录保护。
把抓取结果直接用于写 Shopify。
```

### 场景 2：用户要产品分析

用户说：

```text
帮我分析这个产品有没有机会，顺便看下竞品和趋势。
```

系统发现：

```text
当前任务需要产品机会分析能力。
内部只有基础 Listing Agent，没有专门的 Product Research Skill。
```

系统应该：

```text
1. 先查内部 Skill。
2. 如果没有合适的，再查外部能力目录。
3. 发现类似 Superpower 这种产品分析/研究能力。
4. 展示候选能力的说明、输入输出、边界、数据源、风险。
5. 用户确认后，进入安装/启用申请。
```

关键点：

外部能力可以被推荐，但不能被盲目信任。

必须先回答：

- 它能访问哪些数据？
- 它会不会调用外部 API？
- 它是否会把用户数据传出去？
- 它输出的是建议、报告，还是会执行动作？
- 它是否支持沙盒？
- 它是否有版本和来源？

## 新增核心对象

### CapabilityGap

当 Orchestrator 发现当前能力不够时，先生成能力缺口。

```text
capability_gaps
- id
- tenant_id
- user_id
- conversation_id
- task_goal
- missing_capability_type = skill / tool / mcp / connector / agent / team
- missing_capability_description
- required_inputs
- expected_outputs
- risk_level
- status = detected / searching / resolved / ignored
```

### ExternalCapabilityCandidate

外部候选能力不是马上安装，而是候选。

```text
external_capability_candidates
- id
- source_registry
- external_id
- name
- capability_type
- description
- provider
- homepage
- documentation_url
- install_method
- permissions_requested
- data_access
- network_access
- risk_level
- trust_score
- compatibility_score
- status = discovered / reviewed / rejected / approved_for_install
```

### CapabilityInstallRequest

安装或启用必须有申请。

```text
capability_install_requests
- id
- tenant_id
- requested_by
- conversation_id
- capability_gap_id
- candidate_id
- reason
- requested_permissions
- requested_scopes
- risk_level
- status = pending_user_confirm / pending_admin_review / sandbox_testing / approved / rejected / installed
- reviewed_by
- created_at
- decided_at
```

## 对话中的能力发现流程

```text
1. 用户提出需求。
2. Orchestrator 解析意图。
3. Capability Registry 查内部能力。
4. 如果内部能力不足，生成 CapabilityGap。
5. 根据 gap 查询外部能力目录。
6. 返回 ExternalCapabilityCandidate。
7. Orchestrator 解释候选能力和风险。
8. 用户选择是否申请。
9. 创建 CapabilityInstallRequest。
10. 根据风险进入自动批准、管理员审核或沙盒测试。
11. 通过后安装到租户能力目录。
12. 当前对话可以继续使用，或提示需要刷新/重新运行。
```

## 自动化边界

### 可以自动做

- 自动识别能力缺口。
- 自动搜索候选能力。
- 自动比较内部能力和外部能力。
- 自动生成安装申请。
- 自动生成风险说明。
- 自动建议“用内部能力替代”或“需要外部能力”。
- 对低风险、只读、已审核白名单能力自动申请启用。

### 不能自动做

- 不能无对话上下文后台随机安装。
- 不能静默安装。
- 不能静默赋予租户权限。
- 不能自动启用高风险写工具。
- 不能自动接入会读取用户敏感数据的外部能力。
- 不能自动信任外部 Skill 的输出。
- 不能绕过管理员审核。

## 风险分级

| 风险等级 | 能力类型 | 处理方式 |
| --- | --- | --- |
| low | 纯文本 Skill，无外部网络，无数据外传 | 用户确认后可启用 |
| medium | 只读外部查询 Tool / MCP | 用户确认 + 管理员策略 |
| high | 浏览器自动化、爬虫、读取租户数据 | 管理员审核 + 沙盒测试 |
| critical | 写外部系统、付款、广告预算、发送消息 | 必须人工审核，不允许对话中直接自动安装启用 |

## 候选能力说明必须展示什么

用户或管理员看到的安装确认页必须展示：

```text
能力名称
能力类型
来源
用途
适用场景
不适用场景
需要访问的数据
是否访问外网
是否有副作用
是否会写入外部系统
权限 scopes
风险等级
替代方案
为什么当前任务需要它
```

## 和 Capability Registry 的关系

外部能力安装后，不是直接裸用。

它必须被转换成内部 Capability：

```text
ExternalCapabilityCandidate
  -> CapabilityInstallRequest
  -> Sandbox Evaluation
  -> Internal CapabilityCard
  -> CapabilityVersion
  -> Tenant Enablement
```

也就是说：

```text
外部能力进入平台后，也必须有能力卡片、边界说明、版本、质量指标、权限和审计。
```

## Orchestrator 的话术示例

### 爬虫能力缺口

```text
这个任务需要网页抓取能力。当前你的工作区还没有启用浏览器自动化工具。

我找到了一个候选能力：Playwright MCP。
它适合打开公开网页、抽取页面内容和截图。
它不应该用于绕过登录、验证码、付费墙或网站限制。

是否要为当前工作区申请启用这个能力？
```

### 产品分析能力缺口

```text
你这个需求更像产品机会分析，需要结合竞品、趋势、价格带和评论痛点。

当前内部只有基础 Listing 分析能力，不足以完整覆盖这个任务。
我找到了一个候选产品分析能力：Superpower 类 Product Research Skill。

它可以作为分析辅助，但不会自动执行广告投放或写入店铺。
是否要先申请启用并在沙盒里试跑一次？
```

## 对现有架构的新增要求

### 1. Capability Retrieval 要支持外部源

```text
search_capabilities
  -> internal registry
  -> approved external registry
  -> marketplace index
  -> manually curated list
```

但外部结果默认只是候选，不是可执行能力。

### 2. 安装是工作区级或租户级行为

不要全局自动启用。

```text
tenant_capability_installations
- tenant_id
- capability_id
- enabled_by
- enabled_at
- allowed_roles
- allowed_scopes
- status
```

### 3. 对话要能等待能力安装

如果用户确认安装申请，当前 PlanDraft 可以进入：

```text
waiting_capability_install
```

安装完成后再继续。

### 4. 外部能力必须可撤销

管理员必须能：

- 禁用能力。
- 撤销权限。
- 查看调用日志。
- 查看它被哪些 Agent/Plan/Workflow 使用。
- 回滚版本。

## 最终结论

外部能力发现应该发生在对话中，而不是后台自发安装。

正确模型是：

```text
用户提出需求
  -> AI 发现能力缺口
  -> AI 查找候选外部能力
  -> AI 解释为什么需要
  -> 用户确认申请
  -> 管理员/策略审核
  -> 沙盒测试
  -> 安装成内部 Capability
  -> 当前任务继续执行
```

这样既能让 AI 主动发现好能力，又不会让系统变成不可控的插件自动安装器。
